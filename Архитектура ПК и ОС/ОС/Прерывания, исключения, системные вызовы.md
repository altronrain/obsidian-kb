## Аппаратные прерывания (внешние)
Аппаратные прерывания, с которыми уже знакомились, работают по следующей схеме:
* устройство, которому требуется внимание процессора, устанавливает на шине сигнал "запрос прерывания"
* процессор доводит выполнение текущей программы до такой точки, в которой выполнение можно прервать так, чтобы потом восстановить его с того же места
* процессор выставляет на шине сигнал "подтверждение прерывания", а другие прерывания блокирует
* получив подтверждение, устройство передает по шине некоторое число, идентифицирующее данное устройство (*номер прерывания*)
* процессор сохраняет где-то (например в стеке активной задачи) текущие значения *счетчика команд* и *регистра флагов* (*малое упрятывание*), иначе значения в этих регистрах испортятся и незаметный для первоначальной задачи возврат из обработчика прерываний будет невозможен (остальные регистры может сохранить сам обработчик)
* устанавливается *привилегированный режим* работы CPU
* управление передается на точку входа процедуры ОС, называемой обработчиком прерывания

Так, например, ранее рассматриваемый *таймер* является самым простым внешним устройством. Все что он делает -- подает запросы на прерывание (совершает аппаратное прерывание) через равные промежутки времени

## Исключения. Внутренние прерывания
Если внутри CPU при выполнении активной задачи случается ситуация, когда *процессор не может выполнить операцию/команду*, говорят о том, что возникает *исключение*.
Выполнить операцию процессор не может, завершить задачу сам процессор не может и не в праве. Все что ему остается, это передать управление ОС, известив ее о ситуации.
Для этого требуется:
* переключиться в привилегированный режим работы CPU
*  сохранить где-то (например в стеке активной задачи) текущие значения *счетчика команд* и *регистра флагов*
* передать управление коду ОС

В терминологии i386 исключения называются *внутренними прерываниями*, так как возникают внутри CPU при выполнении задачи, а не приходит извне. А механизм действий CPU схож

*Исключения возникают в рамках выполнения задачи как единицы планирования*, так что задачу вообще некорректно считать прерванной, но даже если допустить это, то выходит что она прервала сама себя, устроив аварию.

Аппаратные прерывания, очевидно, действительно прерывают выполнение текущей задачи.

## Отличия аппаратных прерываний от исключений
Описанные шаги  для исключений с точки зрения CPU по сути своей идентичны выполнению аппаратного прерывания, с той лишь разницей что отсутствует этап с общей шиной и аппаратным устройством. Сама ситуация возникает *внутри* CPU, а не поступает извне

Исключения от аппаратных прерываний отличаются еще и тем, что они возникают только при *наличии* задачи, а задача является *единицей планирования*, то есть ее можно приостановить, а затем продолжить с того же места. Этот момент позволяет ОС выполнять обработку исключений совершенно иначе, чем аппаратных прерываний. 
Обработка исключения происходит *в контексте пользовательской задачи*.

## Схожесть аппаратных прерываний и исключений
В обоих случаях при обработке где-то в памяти должен хранится *обработчик*, на который следует передать управление при наступлении соответствующей ситуации. адрес обработчика должен быть настраиваемым, но его настройка должна быть привилегированным действием (выполняться только ОС).

Для семейства i386 процессоров все обработчики, предназначенные как для исключений так и для аппаратных прерываний, снабжены номерами от `0` до `255`. В оперативной памяти выделяется специальная область для хранения *таблицы дескрипторов прерываний*. Таблица содержит записи по `8` байт, каждая запись соответствует своему обработчику и содержит информацию о том, как именно следует ему передавать управление (по какому адресу, в какой сегмент, следует ли блокировать аппаратные прерывания)

Именно потому что нумерация сквозная, разработчики i386 и ввели понятие внутренних прерываний для исключений
Для других CPU это может быть совершенно не так (аппаратные прерывания -- прерывания, а исключения могут называть исключениями или ловушками).

## Системные вызовы
Пользовательская задача не может делать ничего кроме преобразования данных в отведенной ей области памяти. Все иные действия, затрагивающие внешний по отношению к задаче мир, выполняются через ОС.
Механизм, позволяющий пользовательской задаче обратиться к ядру ОС называется *системным вызовом*.
Для его выполнения необходимо:
* сменить режим выполнения с пользовательского на привилегированный
* передать управление в некоторую точку входа в ОС
эти действия должны происходить по инициативе *пользовательской задачи*, которая:
* работает в ограниченном режиме
* не имеет доступа к пространству адресов ядра
то есть требуется некая специальная команда CPU.

## Программные прерывания
На различных архитектурах CPU такая инструкция может именоваться по разному (`trap, svc, syscall`). Куда будет передано управление, определяет ОС, в которой должен присутствовать некий *обработчик* системных вызовов.

Так как механизм в чем-то схож с прерываниями создатели x86 не стали изобретать отдельного механизма для системных вызовов. В систему команд ввели команду `int` (от interrupt), которая изначально *принудительно* вызывала обработчик прерываний.
Эффект команды `int` назвали *программным прерыванием*
На i386 командой `int` может быть вызван только специально для нее предназначенный обработчик.
Таким образом в x86 процессорах системные вызовы реализуются через программные прерывания

Отличие программного прерывания от исключения в том, что оно происходит по инициативе пользовательской задачи.

Очевидно, как и исключение, программное прерывание точно ничего *не прерывает*.

## Именование сущностей в различных архитектурах CPU
Итого, для x86 процессоров, можно различать 3 вида *прерываний*:
* внешние прерывания (аппаратные)
* внутренние прерывания
* программные прерывания

В иных архитектурах с именованием в целом понятнее:
* прерываниями называются только "настоящие" аппаратные прерывания
* для внутренних прерываний используются термины ловушка или исключение
* для программных прерываний используется термин системный вызов
