Список критериев:
1. Тестирования функций
2. Тестирование классов входных данных
3. Тестирование классов выходных данных
4. Тестирование области допустимых значений (пограничное тестирование)
5. Тестирование длины набора данных
6. Тестирование упорядоченности набора данных

Разберем перечисленных критерии чуть более подробно.
## Критерий тестирования функций

Актуален для многофункциональных программ. Требует подбора такого набора тестов, чтобы был выполнен хотя бы один тест для каждой из функций, реализуемых программой.
## Критерий тестирования классов входных данных

Требует классифицировать множество входных данных на классы с равнозначными наборами данных.
Считаем, что программа правильно отрабатывает на одном наборе входных данных из класса, то она будет работать аналогичным образом на любом ином наборе из указанного класса.
Требует выполнения хотя бы одного теста для каждого класса входных данных.
## Критерий тестирования выходных данных

Аналогично предыдущему пункту, только проверке подвергаются выходные данные.

Три предыдущих пункта хорошо согласуются между собой. Обычно, каждой функции программы соответствует свой набор входных и выходных данных.

## Критерий тестирования области допустимых значений

При рассмотрении области допустимых значений (переменной, входных данных), если диапазон прост и небольшой (*например: пол, ноты*), то достаточно проверить значения диапазона и вне диапазона.

Если рассматриваемый диапазон большой, (*например: числовой диапазон*), то обычно рассматривают следующие виды значений:
1. нормальные условия (в середине диапазона)
2. граничные (экстремальные) условия
3. исключительные условия (выход за границы диапазона)
Интересным может быть случай минимального выхода за границы диапазона рассматриваемого класса.

Следующие два критерия фактически являются частными случаями ранее описанных, но они очень важны.

## Критерий тестирования длины набора данных

Можно рассматривать как частный случай тестирования области допустимых значений. 
Если обрабатывается какой-то набор данных, то имеет смысл проверить:
1. пустой набор данных (без элементов)
2. единичный набор (содержит один элемент)
3. слишком короткий набор (если предусмотрена минимальная длина)
4. набор минимально возможной длины (если предусмотрена)
5. нормальный набор (из нескольких элементов)
6. набор из нескольких частей (если возможно)
7. набор максимально возможной длины (если предусмотрено)
8. слишком длинный набор (с длиной больше максимально допустимой)

## Критерий тестирования упорядоченности данных

Имеет смысл для задач сортировки и поиска экстремумов. Имеет смысл проверить следующие ситуации:
1. данные неупорядочены
2. данные упорядочены в прямом порядке
3. данные упорядочены в обратном порядке
4. в наборе имеются повторяющиеся значения
5. экстремальное значение находится в середине набора
6. экстремальное значение находится в начале набора
7. экстремальное значение находится в конце набора
8. в наборе несколько совпадающих экстремальных значений
