Рассматриваются следующие распространенные методы:
* эквивалентное разбиение
* анализ граничных значений
* таблица решений
* таблица переходов
## Эквивалентное разбиение

Подразумевает деление данных на группы (классы эквивалентности) по какому либо признаку. Метод основывается на предположении, что все элементы одной группы обрабатываются объектом тестирования схожим образом.

Если при выполнении тестового сценария обнаруживается дефект для одного из значений, относящихся к какому-либо классу, тот же дефект должен быть обнаружен и для любых других значений из того же класса. Следовательно, достаточно одного теста на каждый класс.

Классы эквивалентности можно задать для любого элемента данных, относящегося к объекту тестирования. Например:
* входные данные
* выходные данные
* элементы конфигурации
* внутренние значения
* значения с привязкой ко времени
* параметры интерфейса
Классы могут быть:
* непрерывными и дискретными
* упорядоченными и неупорядоченными
* конечными и бесконечными
Классы не должны пересекаться и быть пустыми.

На практике сложно понять, каким образом корректно разбить данные на классы. В пример можно привести позитивные и негативные классы. В одном значения, про которые известно по спецификации, как с ними работать, а во втором, те значения, которые не могут быть обработаны или поведение не определено спецификацией.

100% покрытие достигается проверкой всех классов (в т. ч. негативных). Минимум 1 проверка на класс. Покрытие вычисляется как отношение числа классов, на которых выполнен как минимум один тестовый сценарий, к общему числу заданных классов, выраженное в процентах.

Объекты тестирования часто имеют множественные наборы классов эквивалентности (например со множеством входных параметров). В таком случае тестовый сценарий будет покрывать несколько классов из разных наборов. 
Самый простой подход в таком случае, подобрать тест-кейсы так, чтобы был покрыт каждый вариант.  Такой подход предполагает, что тестовые сценарии выполняются для каждого класса из каждого набора классов по меньшей мере один раз. При этом не берутся во внимание комбинации различных классов.
## Анализ граничных значений

Анализ граничных значений — это метод, в котором проверка выполняется для границ классов эквивалентности. Метод может использоваться для упорядоченных классов.
Граничными значениями класса являются его минимальное и максимальное значения.
Граничные значения важны по той причине, что разработчики с большей вероятностью допускают ошибки именно для них.

Существует два основных варианта упомянутого метода: с определением двух граничных значений и с определением трех граничных значений. Они отличаются количеством приходящихся на каждую границу элементов, для которых необходимо выполнить тест, чтобы достичь 100% покрытия.

В варианте метода с двумя граничными значениями для каждого граничного значения берется два элемента покрытия: само граничное значение и наиболее близкое соседнее значение из смежного класса. Покрытие вычисляется как отношение числа граничных значений, для которых выполнялось тестирование, к общему числу заданных граничных значений, выраженное в процентах.

В варианте метода с тремя граничными значениями для каждого граничного значения берется три элемента покрытия: само граничное значение и два соседних значения.
Покрытие вычисляется как отношение числа значений, граничных и соседних к ним, для которых выполнялось тестирование, к общему числу таких значений, выраженное в процентах.

Более тщательным (но и трудоемким) является вариант с тройками граничных значений, он может обнаружить дефекты, которые не обнаружит метод с двойками. 
Пример на пальцах:
Условие  в спецификации: `if x <=10`
Условие в реализации: `if x = 10`
Метод с двойками: `x=10, x=11` -- проблема не обнаружена
(не частный случай, по спеке именно 11 будет в другом классе)
Метод с тройками: `x=9, x=10, x=11` -- проблема обнаружена

## Таблица решений

Используются для тестирования программной реализации требований к приложению (системе), где последние определяют какие комбинации условий приводят к каким результатам. Таблицы решений — это наглядный способ представления сложных логических правил, например, бизнес-правил

При создании таблицы решений в системе определяются исходные условия и результирующие действия. Эти элементы формируют строки таблицы. Каждый столбец представляет собой правило, задающее уникальную комбинацию условий, а также относящиеся к нему действия

Условия обозначаются следующим образом: 
* «T» («true», истина) -- выполнение условия
* «F» («false», ложь) -- невыполнение условия
* «–» -- содержание условия не оказывает влияния на результирующее действие
* «N/A» («Not Available» — недоступно) -- условие невозможно реализовать для данного правила
* «X» в блоке действий -- должна выдаваться ошибка
* Пустая ячейка -- действие не произойдет. 
 Могут использоваться и другие обозначения.

В тестировании с помощью таблицы решений элементами покрытия являются столбцы, содержащие действительные, выполнимые комбинации условий. Покрытие вычисляется как отношение числа столбцов, для которых выполнялось тестирование, к общему числу столбцов с выполнимыми условиями, выраженное в процентах.

Плюсы:
* метод обеспечивает систематический подход к определению всех возможных комбинаций условий, некоторые из которых в противном случае могли бы быть упущены из виду
* позволяет найти пробелы или противоречия в требованиях
Минусы:
* число правил растет экспоненциально с ростом числа условий (если условий много, от проверка таблицы займет уйму времени)
## Таблица переходов

Диаграмма перехода состояний воспроизводит модель поведения системы, показывая возможные состояния и разрешенные переходы между этими состояниями. Любой переход вызывается событием, которое может к тому же сопровождаться ограничением. Типичный синтаксис, маркирующий переход, следующий: «событие \[ограничение\] / действие».

Таблица переходов — модель, эквивалентная диаграмме перехода состояний. Ее строки представляют собой состояния, а столбцы — события (вместе с контрольными условиями, если таковые имеются). Записи в таблице (ячейки) представляют собой переходы и содержат конечное состояние, а также результирующие действия, если они определены. В отличие от диаграммы состояний и переходов, таблица переходов однозначно отображает недействительные переходы, которые представлены пустыми ячейками.

В тестировании перехода состояний существует много критериев, используемых для задания покрытия. Рассмотрим три из них:
1. При покрытии всех состояний элементами покрытия являются состояния.
2. При покрытии всех разрешенных переходов (или «одношаговом» покрытии) элементами покрытия являются однократные переходы из числа разрешенных.
3. При покрытии всех переходов элементами покрытия являются все переходы, имеющиеся в таблице переходов. Также следует делать попытку недействительных переходов. Тестирование только одного недействительного перехода в рамках каждого теста помогает избежать замаскированных дефектов - ситуации, в которой один дефект мешает обнаружить другой.

Покрытие разрешенных переходов — наиболее распространенный критерий покрытия. Обеспечение полного покрытия разрешенных переходов гарантирует и полное покрытие состояний. Обеспечение полного покрытия переходов гарантирует как полное покрытие состояний, так и полное покрытие разрешенных переходов.
