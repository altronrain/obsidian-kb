Определение места ошибки.

Прежде всего следует различать место *проявления* ошибки и место *нахождения* ошибки.

На примере фрагмента кода
```
a:= 0;
if x>3 then a:= 10;
b:= 1/a;
```
место проявления ошибки -- третий оператор фрагмента `b:= 1/a`
место нахождения ошибки так просто не назовешь, зависит от постановки задачи.

Место проявления ошибки -- можно определить по внешним признакам.
Место нахождения ошибки -- определяется путем содержательного анализа теста программы.
## Отладочные операторы

Обычно, для облегчения процесса поиска ошибок, в системах программирования имеются специальные отладочные средства.  Помимо них часто можно использовать отладочные операторы, добавляемые в текст (вспоминаем pdb Python).

Для проверки условий/значений переменных и т.д.  в определенных местах программ можно использовать конструкции проверки `assert <condition>` (в Python и pytest так тесты и пишутся)
## Индуктивный и дедуктивный методы поиска ошибки

Это два разных подхода к поиску ошибок.

Индуктивный подход -- движение от частного к общему. От того, какие данные говорят об ошибке, к тому, как их можно объяснить. 

Ход рассуждений при индуктивном подходе:
```
Сбор данных о работе программы <--------------------|
|                                                   |
Выделение данных, имеющих отношение к ошибке <------|
|                                                   |
Изучение зависимостей между этими данными <---------|
|                                                   |
Выдвижение гипотезы об ошибке <---------------------|
|													|
Проверка гипотезы                                   |
|													|
Гипотеза подтвердилать? -> Нет ---------------------|
|
Да
|
Нахождение ошибки
```
\
Дедуктивный подход -- движение от общего к частному. Что в принципе могло произойти? Исходя из некоторых соображений о работе программы формируется множество гипотез. Затем они уточняются, отсеиваются, как несоответствующие признакам ошибки.
Имеет свои плюсы с накоплением опыта.

Ход рассуждений при дедуктивном подходе:
```
Перечисление гипотез <------------------------------|
|                                                   |
Исключение и уточнение гипотез                      |
|                                                   |
Исключены все гипотезы? -> Да -> Сбор дополнительных данных
|                                                   |
Нет                                                 |
|													|
Проверка выбранной гипотезы                         |
|													|
Гипотеза подтвердилать? -> Нет ---------------------|
|
Да
|
Нахождение ошибки
```
## Ретроанализ

Дословно, "открутка" действий из текущей ситуации, с целью понять как она сложилась.
Применительно к отладке, находя место проявления ошибки, мы можем знать какие переменные в каких значениях находились. Далее откатываясь по шажкам назад начинаем раскручивать, как эти переменные получили текущие значения, через какие другие переменные, а откуда значения получили они и т.д. 
## Инструментарий отладки

В средах разработки и самих ЯП часто присутствует встроенный инструментарий для отладки. Основным можно считать:
1. Пошаговое выполнение программы (с заходом в процедуры и без захода)
2. Исполнение до курсора
3. Установка контрольных точек (возможно условных и/или со счетчиком)
4. Вывод значений переменных на момент выполнения
5. Наблюдение за состоянием стека вызванных подпрограмм
## Принципы отладки

1. Придется подумать. Средства отладки и отладочные операторы это хорошо, но бездумное их применение даст мало толка.
2. Не прибегать к экспериментированию. Работа по принципу "не знаю где ошибка, сейчас *что-нибудь* поправлю и посмотрю результат" -- недопустима.
3. Исправлять следует поочередно. Внесение *нескольких* исправлений за раз существенно затрудняет анализ последствий.
4. Необходимо найти ошибку, объясняющую все 100% симптомов.
5. Там где есть одна ошибка, могут быть и другие.
6. Исправление может внести новые ошибки.
## Анализ обнаруженной ошибки

Чтобы извлечь пользу на будущее из допущенной ошибки в программе, необходимо ее проанализировать по следующему плану:
1. Когда была сделана ошибка? На каком этапе работы над программой?
   (постановка задачи, проектирование, написание кода)
2. Почему была сделана ошибка? Что именно было непонятно?
   (была ли неточность в формулировке задачи, не было полного понимания конструкций ЯП, неудачный выбор имен переменных)
3. Как можно было предотвратить ошибку? Что в следующий раз надо делать по-другому чтобы подобные ошибки не повторялись?
4. Почему ошибку не обнаружили раньше? Почему не на более ранних этапах?
5. Как можно было обнаружить ошибку раньше? Как ее нашли? Нельзя ли применить успешный тест в других программах?
Полезно фиксировать эту информацию.
