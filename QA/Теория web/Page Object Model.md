## Основные сведения

Парадигма/паттерн программирования, используемый в автотестах веб-продуктов.
Удобный способ структурировать код проекта.

Идея заключается в представлении каждой страницы в виде класса. Классу присущ набор атрибутов и доступных методов. 

Так атрибуты в основной своей массе будут описывать различные элементы на странице.
Методы же будут представлять собой выполнение различных операций, среди которых можно выделить:
* действия пользователя (автоматизация рутинных действий в виде методов, например авторизация), иными словами "сделать что-то"
* выполнение проверок условий/состояний элементов, иными словами "проверить что-то"

В идеале тест с использованием POM должен описывать "бизнес-логику" тестового сценария, скрывая методы фреймворка работы с браузером (таких как Selenium или Playwright). В таком случае при внесении правок в страницу, для адаптации тестов необходимо будет поправить только объект-страницу.

Полезно структурировать такие страницы в отдельной папке проекта автоматизации тестирования.
## Переходы между страницами

При использовании POM возможны два варианта реализации перехода на другие страницы. 
Явный и неявный.

### Явный переход. Неявная инициализация страницы

Явный переход заключается в реализации метода перехода с возвратом экземпляра новой страницы. Делается это в методах описываемой страницы, с которой совершается переход, при этом необходимо импортировать класс страницы, на которую переход совершается.

Пример:
```python
def go_to_dashboard_page(self):
	self.dashboard_link.click()
	return DashboardPage(browser=self.browser, url=self.browser.current_url)
```
При таком подходе не нужно думать об инициализации страниц в теле теста. Мы получаем их экземпляры из метода:
```python
test_user_can_go_to_dashboard(browser, main_page):
	dashboard_page = main_page.go_to_dashboard_page()
	dashboard_page.is_dashboard_page()
```

Плюсы такого подхода:
1. Тесты выглядят аккуратнее. Страницы не нужно инициализировать в теле теста.
2. Потенциально уменьшаем дублирование кода, не во всех тестах нужно инициализировать множество страниц.
Минусы:
1. При увеличении количества страниц растет количество перекрестных методов-переходов
2. При изменении логики работы требуется вносить много правок
3. Страница инициализируется неявно, что может затруднить понимание кода
4. Образуются циклические зависимости в коде (в импортах)
### Неявный переход. Явная инициализация страницы

При неявном переходе метод перехода не возвращает экземпляра новой страницы:

Пример:
```python
def go_to_dashboard_page(self):
	self.dashboard_link.click()
```
Вместо этого страница инициализируется явно в теле теста:
```python
test_user_can_go_to_dashboard(browser, main_page):
	main_page.go_to_dashboard_page()
	dashboard_page = DashboardPage(browser=self.browser, url=self.browser.current_url)
	dashboard_page.is_dashboard_page()
```

Плюсы такого подхода:
1. Меньше циклических зависимостей в коде
2. Меньше импортов
3. Большая гибкость
4. Более понятен код теста, так как страница инициализируется явно
Минусы:
1. Лишний шаг в теле тест-кейса
2. Дублирование кода
3. При написании тест-кейсов нужно думать про корректные переходы

На практике я применяю второй подход, однако он модифицирован за счет фикстур [[PyTest]].
В `conftest.py` я объявляю ряд фикстур вида:
```python
@pytest.fixture()
dashboard_page(page):
	return DashboardPage(page)
```
Что позволяет в объявлении теста указать используемые им страницы, затем в теле теста после перехода работать с новым объектом сразу:
```python
test_user_can_go_to_dashboard(page, main_page, dashboard_page):
	main_page.go_to_dashboard_page()
	dashboard_page.is_dashboard_page()
```
