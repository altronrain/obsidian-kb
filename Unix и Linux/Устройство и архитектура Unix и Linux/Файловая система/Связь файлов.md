Метаданные не содержат имени файла, ни даже указателя на это имя. Только записи в каталогах связывают метаданные (а точнее номер индексного дескриптора, в котором они содержатся) с именем файла.

Сами *данные* (то есть с*ам файл*) существует независимо от того, как его называют в файловой системе. Такая архитектура позволяет одному файлу иметь несколько имен. 

Строго говоря, файлу вообще безразлично, *какие имена* он имеет в файловой системе, но не безразлично *количество* этих *имен*. Если в файловой системе более не осталось имен файлов, ссылающихся на сам файл (данные), то такой файл должен быть удален с диска.

В Unix/Linux имеется два типа связей файлов:
* жесткая связь
* символьная связь

## Жесткая связь
Это связь имени файла с его данными (указание на индексный дексриптор).

Создается командой `ln`:
```
ln first /home/sergey/second
```

![[Pasted image 20260612221147.png]]

Жесткие связи абсолютно равноправны. Это просто два разных имени файла, ссылающихся на одни и те же данные.
* изменения, внесенные в один файл будут видны и в другом, так как изменяются сами данные
* удаление одного из имен файла не приведет к удалению самого файла, пока есть иные имена, ссылающиеся на эти данные

Жесткие связи указывают на один и тот же индексный дескриптор, что можно проверить, выполнив команду `ln` с ключом `-i`.
Информацию о количестве жестких ссылок на данные можно получить, выполнив команду `ln` с ключом `-l` (из второго столбца вывода).

Естественным ограничением жесткой связи является требование расположения на одном физическом носителе. Что логично, так как они ссылаются на индексный дескриптор, завязанный на конкретный диск/устройство.

## Символическая связь
Особый тип файла. Данными подобного файла-ссылки является имя целевого файла (скорее его расположение в файловой системе).

Такой файл никак не связан с индексным дескриптором и метаданными файла-оригинала, так как адресует их *косвенно*.

Создается командой `ln` с ключом `-s`:
```
ln -s first /home/sergey/symfirst
```
