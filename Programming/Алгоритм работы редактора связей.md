Не рассматриваются разделяемые библиотеки (с суффиксом `.so`) с концепцией динамической загрузки

## Передача аргументов для работы
Редактор связей (компоновщик, линкер) принимает ряд аргументов к обработке:
* список объектных файлов
* библиотеки (как по имени файла так и по имени библиотеки через `-l`)
	* без суффикса `.a`
	* без префикса `lib`

При указании библиотеки через флаг `-l` линкер сначала пытается найти соответствующий файл в известном ему списке системных директорий (`/lib, /usr/lib...`), а также в директориях, указанных флагом `-L` (причем указание флагом будет иметь приоритет и просматриваться в первую очередь)

## Начало работы компоновщика
В своей работе редактор связей использует *два списка символов* (меток):
* список известных (разрешенных) символов
* список неразрешенных ссылок

В первый заносятся *экспортируемые* символы из объектных модулей (те, что мы делаем доступными другим, например помеченные как `global` в NASM)
Во второй заносятся символы, к которым есть обращения, то есть *импортирующие* (те. что запрашиваются у других модулей/объектных файлов, например помеченные как `extern` в NASM)

В начале работы список разрешенных символов пустой, а список неразрешенных содержит точку входа в программу `_start`

## Принцип обработки объектных файлов
Линкер перебирает переданный список объектов (объектных файлов), поочередно их обрабатывая:
* заносит экспортируемые модулем символы в список известных
	* если часть из них присутствовала в списке неразрешенных ссылок, они оттуда удаляются
* импортируемые модулем символы заносятся в список неразрешенных ссылок только если к этому моменту они не фигурируют в списке известных символов
* объектный код модуля принимается к преобразованию в исполняемый код

## Принцип обработки библиотек
Принцип обработки библиотек несколько сложнее, так как весьма вероятно, что все модули из нее не потребуются.

Если на момент обнаружения библиотеки среди переданных параметров редактору связей список неразрешенных элементов пуст -- библиотека игнорируется

Если список не пуст:
* модули в составе библиотеки начинают просматриваться на предмет наличия одного или нескольких необходимых символов/имен подпрограмм
* если модуль найден, то компоновщик "принимает" его и модифицирует списки:
	* помещает туда все экспортируемые символы из модуля
	* помещает туда все импортируемые символы из модуля
* переходит к рассмотрению следующих модулей библиотеки

операции циклически повторяются пока *ни один из оставшихся* в библиотеке непринятых модулей *не будет пригоден* для разрешения ссылок

Такая организация работы позволяет гарантировать, что из библиотеки будут взяты только те модули, которые нужны, чтобы:
* удовлетворить потребности предыдущих модулей в списке импортов (в том числе головной программы которая сама есть модуль)
* удовлетворить потребности самих импортируемых из библиотеки модулей

## Ошибки редактора связей
Существует две основные ошибки по итогу работы редактора связей:
* неопределенные ссылки (undefined reference) -- список переданных объектов исчерпался, а в списке неразрешенных имен еще остались записи
* конфликт имен -- два или более принятых к рассмотрению модуля экспортируют один и тот же символ
во втором случае некоторые компиляторы прощают такое поведение, но надеяться на это не стоит

## Свойства редактора связей и принципы построения библиотек
В своей работе редактор связей *никогда не возвращается назад* в своем движении по списку передаваемых объектов. 

Отсюда возникают следующие нюансы, если на момент просмотра списка, какому-то модулю потребуется какой-то еще один модуль из библиотеки, что была отсмотрена, то линковка не пройдет.

В связи с этим можно вывести некоторые правила и принципы:
* библиотеки при линковке всегда стоит указывать последними после объектных файлов
* библиотеки никогда не должны перекрестно зависеть друг от друга
во втором случае стоит подумать об объединении библиотек или разрешить эту ситуацию еще как-нибудь.

Зависимости библиотек между собой порождают нетривиальную проблему их указания в нужном порядке для компоновки исполняемого файла
