Если контекст решаемой задачи предполагает начало работы с пустого списка, то есть списка, не содержащего ни одного элемента, обязательно следует превратить указатель в корректный пустой список, занеся в него "ничего" 
Например, в Pascal: `nil`

## Односвязные списки
Динамическая структура данных, строится из *звеньев*, каждое из которых, представляя собой переменную типа запись, в качестве одного из своих полей имеет *указатель на следующий элемент списка* (поле должно иметь тип "указатель на запись того же типа"). Последний элемент содержит в этом поле значение `nil`

Для обращения с таким списком достаточно в каком-нибудь указателе хранить адрес первого элемента, а далее мы можем добраться до любого элемента списка.

Преимуществом данного списка как динамической структуры данных являетсяего *нефиксированный размер*

*Односвязный* означает, что:
* на каждый из элементов списка указывает *один* указатель
* каждая запись содержит одно поле, предназначенное для сохранения связности списка и содержащая указатель
* в списке выстроена одна цепочка связей из указателей
Все эти утверждения эквивалентны

На практике обычно используют один из двух способов:
* заносят элементы всегда в начало с помощью вспомогательного указателя
* хранят два указателя (на начало и конец списка) и заносят элементы в конец

### Работа при помощи вспомогательного указателя
Решаемые типичные задачи:
* составление односвязного списка с записью элементов в начало (получаем перевернуютую последовательность)

Добавление элемента в начало списка производится в три этапа:
* используя вспомогательный указатель мы создаем *новый элемент списка*
* заполняем новый элемент значением и делаем так, чтобы его *указатель на следующий элемент стал указывать на текущий* (который пока является первым, а станет вторым) -- это адрес текущего указателя
* признание нового элемента первым элементом списка путем занесения его адреса в указатель, хранящий адрес первого элемента
![[Pasted image 20241114205322.png]]
На примере синтаксиса Pascal:
```pascal
    while not SeekEof do
    begin
        new(tmp);
        tmp^.data := {some_data};
        tmp^.next := first;
        first := tmp;
    end;
```

Для случая *пустого списка* *последовательность действий* по добавлению первого элемента *полностью совпадает* с последовательностью действий по добавелнию элемента в начало списка, уже содержащего элементы, при условии что перед началом работы указатель `first` указывает на "ничего" (`nil`).

#### Вывод элементов
При занесении чисел все время в начало, располагаться они будут в обратном порядке
Пример вывода чисел в обратном порядке *без очистки памяти*:
```pascal
tmp := first;
while tmp <> nil do
begin
	writeln(tmp^.data);
	tmp := tmp^.next;
end;
```

Чаще на практике требуется сразу же удалять элементы, высвобождая память. Выполнить это можно несколькими способами:
* Способ 1:
	* запомнить во временный указатель адрес следующего элемента
	* удалить первый элемент
	* указателю на первый элемент присвоить адрес следующего элемента
Пример способа 1 *с очисткой памяти*:
```pascal
while first <> nil do
begin
	tmp := first^.next;
	dispose(first);
	first := tmp;
end;
```

* Способ 2:
	* запомнить адрес первого элемента во вспомогательном указателе
	* исключить этот элемент из списка изменив указатель на первый элемент
	* удалить исключенный первый элемент
Пример способа 2 *с очисткой памяти*:
```pascal
while first <> nil do
begin
	tmp := first;
	first := first^.next;
	dispose(tmp);
end;
```

### Работа с хранением вспомогательного указателя
Решаемые типичные задачи:
* составление односвязного списка с записью элементов в конец (получаемпризнание нового элемента первым элементом списка путем занесения его адреса в указатель, хранящий адрес первого элемента прямую последовательность + можем обратиться/вернуться к первому элементу)

Когда список пуст, оба указателя имеют значение "ничего" `nil`
Добавление элемента в конец списка производится в три этапа:
* создаем *новый элемент списка* в конце (сразу связанный с текущим последним)
* сдвигаем указатель на последний элемент в только что добавленный элемент
* заполняем поля созданного элемента
![[Pasted image 20241114221449.png]]
На примере синтаксиса Pascal:
```pascal
new(last^.next);
last := last^.next;
last^.data := {some_data};
last^.next := nil;
```

Для случая *пустого списка* *последовательность действий* по добавлению первого элемента *является частным случаем*. Ведь `last` никуда не указывает
В таком случае последовательность действий такая:
```pascal
new(first);
last := first;
last^.data := {some_data};
last^.next := nil;
```

Итого, объединяя частный случай и общий с помощью условия получаем:
```pascal
    while not SeekEof do
    begin
        if first = nil then
        begin
	        new(first);
	        last := first
	    end
	    else
	    begin
		    new(last^.next);
		    last := last^.next
		end;
		last^.data := {some_data};
		last^.next := nil;
    end;
```

## Проход по списку указателем на указатель
Решаемые задачи:
* удаление элементов из произвольного места списка
* добавление элементов в произвольное место списка (на первое место или после рассматривамого звена)

При обходе списка, если стоит задача добавить или извлечь элементы по условю, то требуется изменять адрес `.next` из *предыдущего элемента*. При этом частным случаем будет первый элемент.

Для упрощения перебора задействуется *указатель на указатель*
```pascal
pp: ^itemptr;
```
где `itemptr` сам есть указатель, `pp` хранит *адрес указателя на* `item`

При проходе списка `pp` будет *указывать на указатель, в котором содержится адрес текущего рассматриваемого элемента списка*.
![[Pasted image 20241117155706.png]]
Начальное присваивание будет довольно очевидным.
Пример (Pascal):
```pascal
pp := @first;
```
а переход к рассмотрению следующего элемента будет чуть сложнее:
```pascal
pp := @(pp^^.next)
```
`pp^^` -- сам текущий элемент, раз `pp` -- указатель на указатель

Итоговый пример обхода списка указателем на указатель с удалением элементов по условию (отрицательные элементы):
```pascal
pp := @first;
while pp^ <> nil do
begin
	if pp^^.data < 0 then
	begin
		tmp := pp^;
		pp^ := pp^^.next;
		dispose(tmp)
	end
	else
		pp := @(pp^^.next)
end
```
Применение указателя на указатель позволило обощить основной цикл, убрав необходимость в обработке особых случаев.
Пример добавления числа в список отсортированный по возрастанию:
```pascal
pp := @first;
while (pp^ <> nil) and (pp^^.data < n) do {ленивые вычисления}
	pp := @(pp^^.next);
{вставка элемента перед текущим в pp^}
new(tmp);
tmp^.next := pp^;
tmp^.data := n;
pp^ := tmp;
```

## Двусвязные списки
В двусвязном списке:
* на каждый элемент указывает два указателя (из предыдущего элемента и из следующего)
* каждый элемент имеет два указателя (на предыдущий элемент и на следующий)
* в списке присутствуют две цепочки связности -- прямая и обратная

С такими списками часто (но не всегда) работают с помощью двух указателей:
* указатель на первый элемент
* указатель на последний элемент
![[Pasted image 20241117162205.png]]
Для полей указателей обычно используются имена `next` и `prev`.

Двусвязные списки используются реже односвязных, но все же для некоторых задач отлично подходят 
Преимущества двусвязных списков:
* симметрия, позволяющая просматривать список в любом направлении
* зная адрес любого из звеньев двусвязного списка можно найти в памяти все звенья этого списка
Недостатки двусвязных списков:
* требуется вдвое больше действий с указателями при любых операциях вставки и удаления
* сами звенья за счет лишнего поля занимают в памяти больше места

Пример вставок элементов слева и справа, где:
* `first` -- указатель на первый элемент списка
* `last` -- указатель на последний элемент списка
* `current` -- указатель на текущий элемент списка
* `tmp` -- укзатель на добавляемый элемент
Пример вставки элемента слева с обобщением случая вставки в конец (слева от несуществующего):
```pascal
new(tmp);
if current = nil then
	tmp^.prev := last
else
	tmp^.prev := current^.prev;
tmp^.next := current;
tmp^.data := n;
if tmp^.prev = nil then
	first := tmp
else
	tmp^.prev^.next := tmp;
if tmp^.next = nil then
	last := tmp;
else
	tmp^.next^.prev := tmp;
```
Пример вставки элемента справа с обобщением случая вставки в начало (справа от несуществующего):
```pascal
new(tmp);
tmp^.prev := current;
if current = nil then
	tmp^.next := first
else
	tmp^.next := current^.next;
tmp^.data := n;
if tmp^.prev = nil then
	first := tmp
else
	tmp^.prev^.next := tmp;
if tmp^.next = nil then
	last := tmp;
else
	tmp^.next^.prev := tmp;
```

## Иные структуры данных
### Двоичное дерево поиска
Дерево в общем случае имеет преимущество перед списками в скорости поиска

Часто в элементе размещается как информация, по которой проводится поиск (*ключ поиска*), так и результат поиска (какие-то значения). Длительность операции поиска по списку линейно зависит от количества хранящихся записей. Соответственно на больших объемах элементов такой поиск уже не может решить поставленную задачу за приемлемое время. Необходимо использовать другие структуры данных, например *двоичное дерево поиска*.

В дереве записи называются *узлами*. В простейшем случае узел имеет два поля-указателя на левое и на правое поддерево (`left` и `right`). При этом `nil` рассматривается как пустое поддерево.

Двоичное дерево поддерживается в *отсортированном* виде, что означает, что для самого дерева и для любого его поддерева все *значения, меньше* значения в *текущем узле*, располагаются *в левом поддереве*, а все *значения, большие текущего* -- *в правом*.

В таком случае количество операций по поиску элемента пропорционально (равно) *высоте дерева*, то есть длине наибольшей возможной связной цепочки от корня дерева до "листового" узла. При этом дерево высотой *h* *может содержать* $2^h - 1$ узлов.
Пример сбалансированного двоичного дерева поиска:
![[Pasted image 20241117180929.png]]

Но дерево может быть и разбалансированным, содержать меньшее количество элементов при довольно большой высоте. Наихудший вырожденный случай разбалансорованного дерева -- весота равная количеству элементов, что даст время поиска идентичное списку. 

По этому деревья стоит балансировать (довольно сложными алгоритмами, но все же)

### Хеш-таблица
Время поиска для хеш-балиц, в отличие от деревьев и списков вообще не зависит от количества хранимых элементов.

Для создания хеш-таблиц используют хеш-функцию. Это такая арифметическая функция, которая по заданному значению ключа поиска выдаёт целое число (которое должно быть трудно предугадать). При этом на одних и тех же значениях ключа хеш-функция должна выдавать одно и то же число.

Хеш-таблица в свою очередь представляет собой массив размера, заведомо превосходящего нужное количество записей. Изначально все элементы помечаются как свободные. Процесс занесения записи с ключом в массив:
* вычисляется хеш-функция по значению ключа
* вычисля остаток от деления значения хеш-функции на длину массива (обычно длина массива -- простое число)
* если вычисленная позиция пуста -- запись заносят в эту позицию

#### Коллизии и их обработка
Если вычисленная позиция занята другим ключом, такая ситуация называется *коллизией*.
Два метода разрешения коллизий:
* хранить в массиве вместо записей указатели, позволяющие сформировать односвязный список из записей, ключи которых дают одно и то же значение
* использовать для записи *следующую свободную* позицию от вычисленной (при этом при поиске проверяют все ячейки до первой свободной)

Недостатком первого способа является увеличение времени работы за счет обработки списков.
Недостатком второго способа являются нетривиальные алгоритмы удаления записей из таблицы. Примерный алгоритм:
* найти первую свободную запись за удаляемой
* каждый элемент/запись, расположенную между удаленной и первой свободной временно удалить из таблицы, а затем включить обратно по обычному правилу добавления (запись будет включена раньше или в том же месте) 

Оба способа чувствительны к заполнению хеш-таблицы. Просматривая каждую запись мы возвращаемся к линейной зависимости при большом заполнении. На практике считается что хеш-таблица должна быть заполнена не более чем на 2/3 от максимального размера. Если таблица сильно заполнена, то ее нужно перестроить, выбрав максимальной длиной большее простое число. При перестроение таблицы -- очень неэффективная операция, так как каждую запись старой таблицы придется внести в новую по тому же правилу добавления со всеми вычислениями.
