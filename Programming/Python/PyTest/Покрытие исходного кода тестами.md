При проверке кода (См. [[Методы белого ящика]] или [[Критерии белого ящика]]) тестами полезно иметь статистическую информацию о том, какой код выполнялся во время проверок, а какой выполнен не был.

Существует два основных метода измерения покрытия:
* линейное покрытие (line coverage). Соотношение выполненных строк кода к общему количеству строк кода
* покрытие "веток"/решений (branch coverage). Дает информацию о покрытии путей/ветвлений в исходном коде программы

> [!NOTE]
> Стоит иметь ввиду, что иногда покрытие строк может быть большим даже если тесты по исследуемому подмодулю *не* проводились. Это происходит потому что `__init__.py` загружает содержимое файлов, происходит определение функций. Но исполняемый код не вызывается (*содержимое функций*). Но строки есть строки, и потому показывается использование содержимого файла.

Для оценки покрытия кода программы существуют специализированные сторонние [[Модули PyTest]], например `coverage.py` или `pytest-cov`. Первый -- популярный в Python инструмент для вычисления покрытия кода. Второй -- популярный плагин PyTest, с которым вызовы удобнее и короче. Часто используются вместе. Ставятся как отдельные пакеты.

### Запуск с подсчетом покрытия кода
При совместном использовании в запуске PyTest требуется передать путь до исходного кода тестируемой программы (его директорий) или имя установленного пакета при помощи флага `--cov=<path to source or package>` 
Таким образом вызов:
```
pytest --cov=some_src_code_dir test_name
```
эквивалентен вызовам:
```
coverage run --source=some_src_code_dir -m pytest test_name
coverage report
```
Помимо директория с исходным кодом и имени пакета можно также передавать отдельные файлы без расширения и поддиректории.
Можно передавать несколько экземпляров ключа `--cov` при запуске.

### Опции конфигурации coverage
Для `coverage.py` можно создать свой файл конфигурации `.coveragerc` в корневом директории проекта. В качестве одной из полезных фич можем указать эквивалентность имеющегося в поддиректории исходного кода коду установленного пакета:
```
[paths]
source =
	some_project/src/code
	*/site-packages/code
```
Это может быть полезно для повышения читаемости отчета `coverage`, так как в таком случае пути будут короче чем у `venv` установки пакета.

### Отображение пропущенных строк
Для отображения пропущенных строк потребуется сформировать репорт в терминал или в виде HTML.
Для терминала необходимо указать:
```pytest-cov_+_coverage
--cov-report=term-missing
```
```coverage
--show-missing
```
Для HTML:
```pytest-cov_+_coverage
--cov-report=html
```
```coverage
--html
```

### Исключение строк из подсчета покрытия
Для того чтобы исключить ту или иную строку из подсчета необходимо лишь добавить особый комментарий.
Пример
```python
if __name__ == "__main__": # pragma: no cover
```
В таком случае данный блок будет исключен из подсчета. Работает как на одиночные строки так и на блоки. 

### Покрытие тестовых сценариев
Помимо исследования покрытия исходного кода не менее полезно указать для покрытия директорий с тестовыми сценариями (оба можно указать в один запуск).
Это может помочь обнаружить следующие проблемы:
* одинаковые имена тестовых функций
* ошибки копирования и модификации тестов при их разработке
* ошибки маркировки тестовых сценариев (skip *нужных* тестов)
* в целом позволяет оптимизировать код тестовых сценариев
