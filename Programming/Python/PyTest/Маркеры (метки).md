PyTest позволяет маркировать как отдельные тесты, так и классы/группы различными метками с целью изменения поведения с указанными тестами во время работы.

### Встроенные маркеры
В первую очередь рассмотрим встроенные маркеры/метки. 
#### Фильтрация предупреждений в рамках теста
Добавляет фильтр предупреждений для теста
```python
@pytest.mark.filterwarnings(warning)
```
#### Пропуск тестов (безусловный и по условию)
Безусловный пропуск теста. Можно указать причину в атрибуте `reason`
```python
@pytest.mark.skip()

@pytest.mark.skip(reason='Not implemented yet')
```
Пропуск теста по условию. Причину также можно указать.
```python
@pytest.mark.skipif(condition, ..., *, reason)
```
#### Ожидание теста неуспешным
Маркер указывает, что ожидается провал теста при его выполнении
```python
@pytest.mark.xfail(condition, ..., *, reason, run=True, raises=None, srict=xfail_strict)
```
Описание возможных атрибутов:
* `condition` -- условие, при котором срабатывает метка `xfail`
* `reason` -- указание причины
* `run` -- указание о том, выполнять ли тест на самом деле
* `raises` -- указывается исключение или кортеж исключений. позволяет конкретизоровать при каких исключениях срабатывает метка (влияет на статус, `XFAIL` или `FAILED`)
* `strict` -- управляет реакцией на прошедшие тесты. при `strict=False` прошедшие тесты получают статус `XPASS`, при `strict=True` прошедшие тесты получают статус `FAIL`
#### Параметризация тестов
Позволяет вызывать тестовую функцию множество раз с передачей разных значений
```python
@pytest.mark.parametrize(argnames, argvalues, indirect, ids, scope)
```
Смотри раздел [[Параметризация#Параметризация тестовых функций]]
#### Указание применяемых к тесту фикстур
Указывает, что к тесту должны быть применены все указанные фикстуры
```python
@pytest.mark.usefixtures(fixture1, fixture2, ...)
```

#### Основные причины применения skip, skipif и xfail
Объективные причины применения указанных меток:
1. Маркеры `skip` и `skipif` часто применяются с целью разграничения тестов под разное окружение (например ОС или версии ОС)
2. Маркер `xfail` удобен при разработке тестов в парадигме [[TDD]], когда сначала пишутся тестовые сценарии, а затем реализуется функционал.
3. Также `xfail` применяется, когда что-то сломалось (произошла регрессия), но не может быть исправлено быстро. Тогда можно навесить метку `xfail`, указав в `reason` номер задачи

Не стоит разрабатывать тесты под *возможный*, не утвержденный функционал далеко наперед, так как в таком случае нет четкого представления о том, понадобятся ли когда-нибудь эти тесты. Они будут только мешаться.

#### Отображение причин в отдельном блоке вывода
Для вывода указанных в метках причин (`reason`) в отдельном блоке в выводе PyTest используется флаг `-r` с указанием дополнительных значений (статусов тестов)
По умолчанию применяется комбинация `-rfE`, где:
* `f` -- тесты, завершившиеся статусом `FAILED`
* `E` -- тесты, завершившиеся статусом `ERROR`

Чтобы изменить поведение на показ причин для всех тестов, кроме прошедших (`PASSED`) нужно указать значение `a` -- all. Таким образом парамет запуска будет `-ra`
Его также можно указать среди параметров по умолчанию в `pytest.ini`. 
```
[pytest]
addopts =
	-ra
```
См. [[Структура тестового проекта#pytest.ini]]

### Пользовательские маркеры
Собственные метки помогают упорядочить и систематизировать тесты, а также тем самым позволяют запускать отдельные категории тестов на основании метки. См. [[Запуск категорий тестов#Запуск промаркированных тестов]]

Такие метки необходимо регистрировать в `pytest.ini`
См. [[Структура тестового проекта#pytest.ini]]
Синтаксис:
```
[pytest]
markers = 
	marker_name: description
```
По умолчанию PyTest при встрече с незарегистрированной меткой ограничивается выдачей предупреждения. На практике рекомендуется выставлять более строгое поведение, а именно вызов ошибки. Опция выставляется также в `pytest.ini`
```
[pytest]
addopts =
	--strict-markers
```

### Область применения маркеров
Метки (маркеры) можно применять к:
* можно применять на файл с тестами целиком
* можно применять к классу тестов
* можно указывать несколько меток на один тест
* можно указывать отдельные метки для наборов данных в [[Параметризация#Параметризация тестовых функций]]

Метки на класс вешаются аналогично меткам на отдельный тестовый метод.
Несколько меток указываются отдельным декораторами, т. е. в несколько строк.
Метки на файл вешаются указанием в начале файла следующей директивы:
```
pytestmark = pytest.mark.mark_name
```

### Вызов промаркированных тестов
Вызов тестов с определенными метками осуществляется ключом `-m`. См. [[Запуск категорий тестов#Запуск промаркированных тестов]] или [[Краткая справка по ключам запуска]]

### Передача параметров через метки
Помимо функции тегирования и группировки тестовых сценариев метки (маркеры) могут служить для передачи параметров в фикстуру перед тестом.
Пример из реального проекта:
```python
@pytest.mark.config('fau_arp_1_1')
@pytest.mark.node_devices('debian-a1')
def test_fau_arp_1_1(
	self,
	setup,
	et_dev: List[EdgeTelnet],
	nt_dev: List[LinuxTelnet]
):
```
где:
* фикстурой `setup` используется `mark.config`
* фикстурой `nt_dev` используется `mark.node_devices`
* фикстурой `et_dev` используется `mark.edge_devices`, примененной к классу тестов

Код фикстурs модифицируется для извлечения параметров из маркера с помощью `requests`. Для этого используется выражение `request.node.get_closest_marker()` 
Пример: 
```python
...
# Обработка файла конфигурации. Получение списка команд
m = request.node.get_closest_marker('config')
if m and len(m.args) > 0:
	filename = m.args[0]
...
```
В качестве значения метода `get_closest_marker()` передается имя метки
