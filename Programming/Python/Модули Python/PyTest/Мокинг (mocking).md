Mock используется для замены частей системы, чтобы изолировать фрагменты кода тестируемого приложения от остальной системы.
Mock входит в состав пакета `unittest`, по этому для его использования необходимо выполнить следующий импорт:
```python
from unittest import mock
```
Мокингом можно подменять:
* значения атрибутов
* возвращаемые значения вызываемых функций
* целые объекты-классы или методы классов

### Мокинг атрибута (подмена атрибута) (return_value)
Мок имеет несколько различных `patch` методов. Для подмены атрибута подходит/используется `patch.object`. Изменяемый объект должен быть импортирован.
Пример подмены атрибута при тестировании CLI (на Typer):
```python
from unittest import mock
import cards
import pytest
from cards.cli import app
from typer.testing import CliRunner

runner = CliRunner()

def test_mock_version():
	with mock.patch.object(cards '__version__', '1.2.3'):
		result = runner.invoke(app, ['version'])
		assert result.stdout.rstrip() == '1.2.3'
```
Обычно выполняется в менеджере контекста, что гарантирует очистку изменений по выходу из него.

### Мокинг классов и методов (return_value)
В описанном примере будет подменяться объект `cards.CardsDB` и его метод `path()`
Пример подмены класса:
```python
def test_mock_CardsDB():
	with mock.patch.object(cards, "CardsDB") as MockCardsDB:
		print()
		print(f" class:{MockCardsDB}")
		print(f"return_value:{MockCardsDB.return_value}")
		with cards.cli.cards_db() as db:
			print(f" object:{db}")
```
Результат выполнения теста:
```
$ pytest -v -s test_mock.py::test_mock_CardsDB
======================== test session starts =========================
collected 1 item
test_mock.py::test_mock_CardsDB
		 class: <MagicMock name='CardsDB' id='140410645302384'>
  return_value: <MagicMock name='CardsDB()' id='140410647097840'>
		object: <MagicMock name='CardsDB()' id='140410647097840'>
PASSED
========================= 1 passed in 0.03s ==========================
```
На что следует обратить внимание:
* если кто-то вызывает мок-объект, то возвращается *новый экземпляр класса* (новый мок-объект). Как, собственно, обычно и происходит в Python.
* возвращаемый (новый) мок-объект доступен через атрибут `return_value` родительского объекта (оригинального мок-объекта). Это очень удобно.
Соответственно, в примере выше когда работаем в менеджере контекста вызывая `cards.cli.cards_db()`, вместо экземпляра `cards.CardsDB` мы получаем мок-объект, который доступен через `return_value` оригинального мок-объекта (в результате видет одинаковый `id`)

В полученном экземпляре мок-объекта уже можем подменять методы. В примере ниже мы не изменяем сам атрибут `path` но меняем поведение, когда *кто-то его вызывает*. Так что тут снова можно использовать `return_value`.
Пример подмены метода класса (изменения его поведения):
```python
def test_mock_path():
	with mock.patch.object(cards, "CardsDB") as MockCardsDB:
		MockCardsDB.return_value.path.return_value = "/foo/"
		with cards.cli.cards_db() as db:
			print()
			print(f"{db.path=}")
			print(f"{db.path()=}")
```
Результат выполнения теста:
```
$ pytest -v -s test_mock.py::test_mock_path
======================== test session starts =========================
collected 1 item
test_mock.py::test_mock_path
db.path=<MagicMock name='CardsDB().path' id='140712512496016'>
db.path()='/foo/'
PASSED
========================= 1 passed in 0.03s ==========================
```
Метод `path()` вернул значение, которое прописали в `path.return_value` экземпляра мок-объекта.
### Мок-объект в виде фикстуры
Подобные операции из примеров вероятно будут требоваться в тестовых сценариях довольно часто. Потому удобно поместить получение экземпляра мок-объекта в [[Фикстуры в PyTest]].
Пример:
```python
@pytest.fixture()
def mock_cardsdb():
	with mock.patch.object(cards, "CardsDB", autospec=True) as CardsDB:
		yield CardsDB.return_value
```
В теле тестового сценария тогда можно сразу обращаться к фикстуре:
```python
mock_cardsdb.path.return_value = '/foo/'
```
### Синхронизация мок-объекта и подменяемого объекта
По умолчанию создаваемый мок-объект принимает все что угодно, любые методы с любым набором параметров. В подавляющем большинстве случаев этого не требуется, так как стоит задача сэмулировать реальный объект. Ожидается обработка только методов и параметров, присущих подменяемому объекту.
Для достижения поведения, присущего эмулируемому объекту необходимо добавить атрибут `autospec=True` в вызов `path.object`:
```python
def test_good_mock():
	with mock.patch.object(cards, "CardsDB", autospec=True) as CardsDB:
```
Эта небольшая доработка позволяет избежать множества ошибок (и детектировать их), которые в противном случае останутся незамеченными:
* грамматические ошибки в написании имен методов
* изменение параметров API без адаптации кода CLI
* изменение имени метода во время рефакторинга. но не везде

Стоит использовать `autospec=True` всегда, где это возможно. Единственный вариант, когда нельзя использовать его -- это подмена динамических объектов, для которых новые атрибуты и методы добавляются/определяются во время работы (в runtime).

### Мокинг без return_value. Проверка корректного вызова
До текущего момента все применения мокинга подменяли `return_value` атрибутов, классов и методов.
Если подменяемый метод ничего не возвращает, мокинг позволяет получить информацию о том, был ли метод вызван корректно, как ожидалось. 
Так, например, после вызова CLI команды по добавлению записи в DB можем проверять не наличие объекта в DB а корректность вызова API метода.
Пример:
```python
def test_add_with_owner(mock_cardsdb):
	cards_cli("add some task -o brian")
	expected = cards.Card("some task", owner="brian", state="todo")
	mock_cardsdb.add_card.assert_called_with(expected)
```
где сначала идет CLI вызов, который использует `db.add_card()`, затем указываются параметры ожидаемой записи в DB, после чего выполняется сравнение методом `assert_called_with` параметров вызова `add_card`.
Если метод `add_card()` не вызывался, или был вызван с некорректным содержимым, то тест не пройдет.

### Проверка обработки ошибок и ошибочных ситуаций
Помимо успешных корректных операций требуется проверять также заложенную обработку ошибок.
Если какой-то вызываемый метод оригинального объекта генерирует исключение, то его можно проверить указав для рассматриваемого метода мок-объекта атрибут `side_effect` с указанием исключения.
```python
def test_delete_invalid(mock_cardsdb):
	mock_cardsdb.delete_card.side_effect = cards.api.InvalidCardId
	out = cards_cli("delete 25")
	assert "Error: Invalid card id 25" in out
```
Для `delece_card` метода указали исключение в `side_effect` 
Вызвали `cards_cli`, который далее вызовет мок-объект и его метод `delete_card`. После чего эмулируется исключение, что и проверяется в `assert`.

### Плюсы и минусы мокинга
Минусы:
Используя мокинг мы более не проверяем *поведение* (behavior) программы, мы проверяем ее *реализацию* (implementation). Тестирование реализации может быть опасным и времязатратным. Любой рефакторинг (например смена имени переменной) может сломать тесты (если эта переменная подвергалась мокингу). Получается этакая группа тестов, являющихся детекторами изменений в программе, которые очень затратно поддерживать.

Плюсы:
Упрощает проверку, когда развертывание инфраструктуры для тестирования поведения крайне сложно, или неуместно (например тестирование платежных систем банкинга).
