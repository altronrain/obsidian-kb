Рассмотрим продвинутые техники параметризации.
### Структуры данных и объекты как значения параметров
[[Параметризация]] может производиться не только для одного или нескольких параметров в виде строк или чисел. В качестве значений могут выступать структуры данных или объекты целиком.
Пример параметризации в виде объекта:
```python
@pytest.mark.parametrize(
	"starting_card",
	[
		Card("foo", state="todo"),
		Card("foo", state="in prog"),
		Card("foo", state="done"),
	],
)
```
При работе с подобными объектами PyTets по умолчанию не знает как их корректно называть/идентифицировать в итоговом отчете.
Пример отчета:
```
========================= test session starts ========================== collected 3 items
test_ids.py::test_card[starting_card0] PASSED [ 33%] test_ids.py::test_card[starting_card2] PASSED [ 66%] test_ids.py::test_card[starting_card1] PASSED [100%]
========================== 3 passed in 0.07s ===========================
```
PyTest указал в качестве `id` передаваемых объектов значения `starting_cartX`, что для пользователя в отчете не несет смысловой нагрузки. Возникает вопрос удобной для человека дифференциации передаваемых объектов. Для этого есть несколько способов:
* Использование параметра `ids`
* Использование `pytest.param`
* Использование списков ID
#### Указание идентификаторов через параметр ids
Можно указать функцию для генерации идентификаторов используя параметр `ids`.
Часто могут подойти встроенные магические методы `str` или `repr`
Пример указания параметра `ids` как встроенный метод `str`:
```python
card_list = [
	Card("foo", state="todo"),
	Card("foo", state="in prog"),
	Card("foo", state="done"),
]
@pytest.mark.parametrize("starting_card", card_list, ids=str)
def test_id_str(cards_db, starting_card):
```
Результат:
```
========================= test session starts ========================== collected 3 items
test_ids.py::test_id_str[Card(summary='foo', owner=None, state='todo', id=None)] PASSED [ 33%]
test_ids.py::test_id_str[Card(summary='foo', owner=None, state='in prog', id=None)] PASSED [ 66%]
test_ids.py::test_id_str[Card(summary='foo', owner=None, state='done', id=None)] PASSED [100%]
========================== 3 passed in 0.01s ===========================
```
Здесь для выбранного объекта `Cards` результат оказался не очень удачным. 
Можно попробовать указать свою пользовательскую функцию в `ids`
Пример указания пользовательской функции в `ids`:
```python
def card_state(card):
	return card.state

@pytest.mark.parametrize("starting_card", card_list, ids=card_state)
def test_id_func(cards_db, starting_card):
```
Результат:
```
========================= test session starts ========================== collected 3 items test_ids.py::test_id_func[todo] PASSED [ 33%]
test_ids.py::test_id_func[in prog] PASSED [ 66%]
test_ids.py::test_id_func[done] PASSED [100%]
========================== 3 passed in 0.02s ===========================
```
Так как подобные функции чаще всего короткие и одноразового применения, можно обернуть их как [[лямбда-функции]].
Пример указания лямбда-функции в `ids`:
```python
@pytest.mark.parametrize(
	"starting_card", card_list, ids=lambda c: c.state
) 
def test_id_lambda(cards_db, starting_card):
```
Результат будет аналогичным предыдущему примеру.

Параметр `ids` доступен при параметризации фикстур и при использовании `pyterst_generate_tests`.
#### Указание идентификаторов через pytest.param
В разделе [[Параметризация]] уже рассматривали пример, когда через `pytets.param` для определенных наборов данных указывали метки. Конструкция `pytest.param` позволяет указывать параметр`id` для набора тестовых данных.
Пример:
```python
c_list = [
	Card("foo", state="todo"),
	pytest.param(Card("foo", state="in prog"), id="special"),
	Card("foo", state="done"),
]

@pytest.mark.parametrize("starting_card", c_list, ids=lambda c: c.state)
def test_id_param(cards_db, starting_card):
```
Результат:
```
========================= test session starts ========================== collected 3 items
test_ids.py::test_id_param[todo] PASSED [ 33%] test_ids.py::test_id_param[special] PASSED [ 66%] test_ids.py::test_id_param[done] PASSED [100%]
========================== 3 passed in 0.02s ===========================
```
Можно обратить внимание, что приоритет параметра `id` из конструкции `pytest.param` выше чем `ids`. Это полезно в случае, когда несколько наборов тестовых данных нуждаются в специальной обработке.
#### Указание идентификаторов через список ID
Самый простой, но при этом относительно небезопасный вариант передачи идентификаторов. В данном случае вместо функции в параметр `ids` передается список ID.
Пример:
```python
id_list = ["todo", "in prog", "done"]

@pytest.mark.parametrize("starting_card", card_list, ids=id_list)
def test_id_list(cards_db, starting_card):
```
Опасность заключается в том, что нужно быть внимательным, и не забывать обновлять список при обновлении набора данных.
Чуть улучшенный вариант -- использовать словарь, где ключами являются идентификаторы, а значениями сами передаваемые наборы параметров.
Пример:
```python
test_data_dict = {
	'single char': 'x',
	'chars and spaces': 'x y z'
	'mixed case': 'SoMe FaNcY TeXt'
	...
}
@pytest.mark.parametrize(
	'variant', test_data_dict.values(), ids=test_data_dict.keys()
)
def test_summary_variants(cards_db, variant):
```
В общем и целом списки и словари используются, когда сделать функцию для получения ID представляется сложной и нетривиальной задачей.

### Параметризация динамически генерируемыми значениями
Вместо наборов значений для указанного параметра можно передать заранее определенную функцию, которая будет динамически генерировать значения для теста.
Пример:
```python
def test_data(): 
	variants = {
		'single char': 'x',
		'chars and spaces': 'x y z'
		'mixed case': 'SoMe FaNcY TeXt'
		...
	}
	for key, value in variants.items()
		yield pytest.param(value, id=key)

@pytest.mark.parametrize('variant', test_data())
def test_summary_variants(cards_db, variant):
```
Пример, конечно, надуманный, но демонстрирует подход с применением функции генератора.

### Непрямая параметризация (Indirect Parametrization)
Непрямым (indirect) параметром называется тот, который сначала передается в фикстуру, прежде чем быть направленным в тестовую функцию. Эта техника позволяет выполнить какие-либо действия в зависимости от значения параметра. Подход сильно перекликается с [[Параметризация#Параметризация фикстур]]

Для передачи параметров в фикстуру в параметризации используется ключ `indirect=` в котором указываются параметры в виде списка (что один, что несколько). Также можно указать значение `True` для передачи всех возможных параметров в фикстуру.
Пример:
```python
@pytest.mark.parametrize(
	"user", ["admin", "team_member", "visitor"], indirect=["user"]
)
def test_access_rights(user):
```
При этом должна быть и фикстура `user`:
```python
@pytest.fixture()
def user(request):
	role = request.param
	print(f"\nLog in as {role}")
	yield role
	print(f"\nLog out {role}")
```
Фикстура получает значения благодаря конструкции `request.param`, аналогично параметризации фикстур.

При этом оба метода (непрямую параметризацию и параметризацию фикстур) можно совмещать. Так например можно указывать для теста определенный вариант/набор из всех имеющихся.
Пример:
```python
@pytest.fixture(params=["admin", "team_member", "visitor"])
def user(request):
...

def test_everyone(user):
...

@pytest.mark.parametrize("user", ["admin"], indirect=["user"])
def test_just_admin(user):
...
```
Результат выполнения тестов:
```
========================= test session starts ========================== collected 4 items
test_subset.py::test_everyone[admin] PASSED [ 25%] test_subset.py::test_everyone[team_member] PASSED [ 50%] test_subset.py::test_everyone[visitor] PASSED [ 75%] test_subset.py::test_just_admin[admin] PASSED [100%]
========================== 5 passed in 0.01s ===========================
```
Для теста `test_everyone` были выполнены все варианты из параметризированной фикстуры, а для теста `test_just_admin` только вариант `admin`, который был передан через `indirect`.

Схожим образом можно делать опциональную непрямую параметризацию через фикстуру.
Пример:
```python
@pytest.fixture()
def user(request):
	role = getattr(request, "param", "visitor")
	print(f"\nLog in as {role}")
	return role

def test_unspecified_user(user):
...

@pytest.mark.parametrize(
	"user", ["admin", "team_member"], indirect=["user"]
)
def test_admin_and_team_member(user):
...
```
Когда фикстуре не передано параметров, то используется некое значение по умолчанию, если же параметр передан, то используется он (собственно это выполняет выражение `getattr(request, 'param', 'visitor')`.
Результат выполнения тестов:
```
========================= test session starts ========================== collected 3 items
test_optional.py::test_unspecified_user Log in as visitor PASSED test_optional.py::test_admin_and_team_member[admin] Log in as admin PASSED test_optional.py::test_admin_and_team_member[team_member] Log in as team_member PASSED
========================== 3 passed in 0.01s ===========================
```

Параметр `indirect` можно использовать также с `pytest_generate_tests`

Пример `indirect` из реального проекта:
```python
@pytest.mark.parametrize('locator_by_name, cli_type',
[
	pytest.param(
		'fm_ethernet_locator', 'ether', marks=pytest.mark.xfail(
			reason='ethm не считается физическим интерфейсом')),
	('fm_vlan_locator', 'vlan'),
	('fm_bonding_locator', 'bond'),
	('fm_bridge_locator', 'bridge'),
	pytest.param(
		'fm_openvpn_locator', 'ovpn', marks=pytest.mark.skip(
			reason='Пока нет ovpn интерфейсов на ноде'))
], indirect=['locator_by_name']
)
def test_should_show_ifaces_on_filter(
	self,
	setup,
	ifaces_page: InterfacesPage,
	locator_by_name: Locator,
	cli_type: str,
	et_dev: EdgeTelnet
)-> None:
```
