Система сборки `bitbake` написана на Python и является системой сборки, активно используемой OpenEmbedded и Yocto Project.

Во время своей работы `bitbake` оперирует следующими типами файлов:
* `.bb` -- файл рецепта;
* `.bbappend` -- дополнение к файлу рецепта;
* `.conf` -- файл конфигурации;
* `.inc` -- нижележащие файлы дополнений к конфигурации;
* `.bbclass` -- файлы классов.

Кратко рассмотрим большинство из них далее.

## Файлы рецептов `.bb`
Самые базовые файлы, содержащие метаданные. Файлы рецептов предоставляют `bitbake` следующую информацию:
* описание пакета (автор, домашняя страница, лицензия и т. д.)
* версия рецепта
* существующие зависимости (для сборки, а также времени запуска)
* информация о расположении исходного кода
* информация о патчах к исходному коду
* как конфигурировать и компилировать исходный код
* как собрать сгенерированные артефакты в установочные пакеты
* как и куда установить собранный пакет

## Файлы конфигурации `.conf`
Файлы конфигурации определяют различные переменные, управляющие процессом сборки. Содержимое может варьироваться, так например файлы могут определять:
* конфигурацию машины
* параметры дистрибутива
* "тюнинг" компилятора
* общие параметры конфигурации
* конфигурацию пользователя

Основной файл конфигурации называется `bitbake.conf` и располагается в директории `conf`

## Файлы классов `.bbclass`
Классы содержат информацию, которой полезно поделиться между файлами метаданных. Изначально существует один встроенный файл класса -- `base.bbclass` из директория `classes`.  Данный файл *всегда* включен для всех рецептов и классов. Данный файл содержит описание *базовых задач*, таких как:
* fetching
* unpacking
* configuring (empty by default)
* compiling (default is run Makefile)
* installing (empty by default)
* packaging (empty by default)

Перечень операций может быть переопределен или расширен другими классами

## Файлы дополнений `.bbappend`
Данные файлы служат для расширения или переопределения информации, содержащейся в существующем файле рецепта. Подразумевается, что для каждого файла дополнения существует начальный файл рецепта. При этом их имя до суффикса должно совпадать:
```
some_file.bb    some_file.bbappend
```
Исключение составляет использование символа `%` в имени файла дополнения. Он представляет собой маску, и ввиду существующих ограничений, должен располагаться непосредственно перед суффиксом `.bbappend`
Например:
```
some_file_1.21.%.bbappend
```
будет применен к следующему набору файлов:
```
some_file_1.21.1.bb
some_file_1.21.2.bb
some_file_1.21.3.bb
```
но, не к:
```
some_file_1.22.1.bb
```
Чтобы применить файл дополнения к любой версии, достаточно именовать его как:
```
some_file_%.bbappend
```

## Примеры

### Сборка отдельного файла
Сборка отдельного файла рецепта (на практике не применяется):
```
bitbake -b foo_1.0.bb
```
Такая сборка не учитывает никакие зависимости. Может использоваться для дебага.

### Сборка пакета
Сборка пакета:
```
bitbake foo
```
По умолчанию без аргументов `bitbake` принимает только `PROVIDES`. По умолчанию файл рецепта в качестве `PROVIDES` передает свое имя пакета `packagename`. Такая сборка подразумевает обработку зависимостей собираемого пакета.

### Выполнение разных задач для разных тергетов
Когда требуется указать разные действия для разных целей (таргетов), используется следующий синтаксис:
```
bitbake first_recipe:task_A fecond_recipe:task_B
```

### Сборка мультиконфигураций
bitbake может собирать несколько итоговых образов или пакетов одной командой, где разные цели требуют различных конфигураций. 

Для выполнения такой сборки необходимо задать для каждой цели свою конфигурацию отдельно, в специфичном каталоге `multiconfig`, который будет подкаталогом `conf`. В таком случае к целям (таргетам) применяется понятие "мультиконфигурация"
![[Pasted image 20250626215755.png]]
Минимально, подобный файл конфигурации для определенной цели должен указывать машину. на которой производится сборка, а также временный директорий сборки (чаще всего, они не должны пересекаться между мультиконфигурациями)

В завершении, необходимо в каталоге `conf` создать файл `local.conf` с определением переменной `BBMULTICONFIG`, чтобы разрешить мультиконфигурационную сборку. 
Так, для примера выше, имея отдельные мультиконфигурации `target1` и `target2`, в значение переменной необходимо внести:
```
BBMULTICONFIG = "target1 target2"
```

Пример запуска сборки пакета `some_package` для основного образа и дополнительно еще двух мультиконфигураций:
```
bitbake mc::some_package mc:target1:some_package mc:target2:some_package
```

### Определение зависимостей между мультиконфигурациями
Иногда зависимости могут существовать между мультиконфигурациями. Например, чтобы собрать образ для определенной архитектуры, рутовая система другого билда для другой архитектуры уже должна существовать.

Чтобы разрешить зависимости в мультиконфигурационных сборках, их нужно указать в рецепте, используя следующий синтаксис:
```
task_or_package[mcdepends] = "mc:from_multiconfig:to_multiconfig:recipe_name:task_on_which_to_depend"
```
Так, имея мультиконфигурации `target1` и `target2`, директива может выглядеть как:
```
image_task[mcdepends] = "mc:target1:target2:image2:rootfs_task"
```

В таком случае, при выполнении сборки `target1` командой:
```
bitbake mc:target1:image1
```
`bitbake` сам выполнит все шаги по сборке, включая необходимые промежуточные шаги над `target2`

Чаще на таких промежуточных шагах как `rootfs` не останавливаются, а все же сразу собирают образ из зависимости целиком, а затем продолжают сборку первого образа:
```
image_task[mcdepends] = "mc:target1:target2:image2:image_task"
```
