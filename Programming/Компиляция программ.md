Материал по большей части относится к кормпилируемым ЯП
Процесс компиляции зависит от ЯП и используемого компилятора, но в общем случае процесс построения исполняемого файла программы из кода можно разбить на два этапа. Представлено краткое описание процесса.

## Первый этап: построение объектных модулей
Объектный модуль еще не является исполняемым файлом. Он содержит объектный код, которому для превращения в машинный код требуется еще одна стадия обработки.
Объектные файлы формируются и для каждого используемого модуля или библиотеки отдельно.
Объектный файл (модуль) пока все еще содержит метку вместо адресов и точек подключения библиотек и других модулей. Их необходимо собрать все вместе.

Пример создания объектного модуля программы на NASM:
```shell
nasm -f elf hello5.asm
```
где указывается, что на выходе ожидаем объектный файл в формате ELF

Аналогичная ситуация при создании множества объектных файлов:
```nasm
nasm -f elf -dOS_LINUX strlen.asm
nasm -f elf -dOS_LINUX putstr.asm
nasm -f elf -dOS_LINUX getstr.asm
nasm -f elf -dOS_LINUX quit.asm
nasm -f elf -dOS_LINUX greet.asm
```

## Второй этап: компоновка объектных файлов
Подробнее см. [[Алгоритм работы редактора связей]]

Для получения исполняемого файла необходимо соединить вместе все объектные файлы модулей, подключить к ним библиотеки, расставить все ссылки (адреса), тем самым превратив объектный код в машинный. Это задача *компоновщика* (редактора связей, линкера)

Отсутствие сторонних модулей и библиотек не исключает стадии компоновки программ

Пример создания 32-битного исполняемого файла из объектного файла:
```shell
ld -m elf_i386 hello5.o -o hello5
```
где ключ:
* `-m` -- указывает архитектуру
* `-o`  -- указывает имя исполняемого файла (от output)

Пример вызова компоновщика для объединения нескольких объектных файлов в один бинарь:
```shell
ld -m elf_i386 greet.o strlen.o getstr.o putstr.o quit.o -o greet
```

Еще раз: Объектный код, который получается в результате ассемблирования, представляет собой "полуфабрикат" машинного кода, в котором вместо *абсолютных адресов* находится информация о том, *как эти адреса вычислить* и в *какие места кода их следует расставить*.

Все потому что ассемблер в принципе не может вычислить адреса, когда мы имеем дело с модульной структурой, он не имеет понятия где будут расположены в памяти те или иные данные/процедуры.

Для просмотра символов в объектных файлах можно использовать программу `nm` (см. [[nm]])
