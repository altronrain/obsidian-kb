Наличие в одном выражении нескольких побочных эффектов ставит вопрос о том, в какой последовательности эти эффекты произойдут. В языке C последовательность вычислений зачастую бывает не определена, так как компилятор может "отложить" их вычисление из своих собственных соображений.

По этому общей рекомендацией является *минимизировать количество побочных эффектов в выражениях* (стараться, чтобы их было не более одного)
## Неопределенность последовательности вычислений
Наличие в одном выражении нескольких побочных эффектов ставит вопрос о том, в какой последовательности эти эффекты произойдут. В языке C последовательность вычислений зачастую бывает не определена.
Так, например, для функции:
```
f(h1(), h2(), h3())
```
* однозначно можно заявить, что функция `f` будет вызвана последней (после вычисления `h1, h2, h3`)
* порядок вычисления функций `h1, h2, h3` *не определен* (может быть выполнен *в любом порядке*)

Помимо перечисления, то же верно для сложения, например:
```
f()+g()+h()
```

## Неопределенность момента записи переменной
В продолжение неопределенности последовательности вычислений. Для операций, изменяющих значение переменной:
* обычное присваивание
* "хитрые" присваивания вроде `+=` или `&=`
* инкремент
* декремент
не определен момент записи нового значения переменной в память

Пример:
```c
a[i] = i++;
```
Так, если переменная `i` имела значение `7`, то оно и будет записано в элемент массива (см. [[Инкремент и декремент]] про суффиксную форму)
Однако, *не определено в какой элемент массива будет произведена запись* -- в седьмой или в восьмой
Компилятор волен записать в память, занятую переменной `i`, ее новое значение как *до* вычисления `a[i]`, так и *после*

Пример:
```c
int a, b;
a = 1;
b = (a+=5) + (a*=2);
```
тут вообще гора вариантов:
* `6` + `12` -> `18`
* `7` + `2` -> `9`
* у автора получилось `12` + `12` -> `24`

> [!NOTE] Как так получилось
> *Формально* результат присваивания равен значению переменной, стоявшей в нем слева. 
>  * Присваивания прошли последовательно в "естественном" порядке
> * Значение присваиваний как выражений компилятор никуда, кроме переменной `a` заносить не стал
> * Перед сложением обратился дважды к `a`, а там соответственно оказалось одно и то же (результат последнего присваивания)
> 
> Итог: `12` + `12` -> `24`  

## Точки следования
*Точки следования* (sequence points) помогают в какой-то мере справиться с проблемой неопределенной последовательности вычислений 
(могут также называться *точки гарантированных вычислений*)

В каждой такой точке компилятор доводит до конца все побочные эффекты, которые он мог ранее "отложить"
К подобным точкам следования относятся:
* конец выражения:
	* оператор вычисления выражения -- `;`
	  дошли до `;` -- все, что должно было случиться, случилось
	* все выражения, предполагаемые синтаксисом *любых* операторов:
		* условные выражения в `if, while, do-while`
		* выражение в заголовке `switch`
		* выражение, задающее возвращаемое значение в операторе `return`
		* все три выражения в заголовке `for` (каждое по отдельности)
* точки следования в рамках одного выражения:
	* логические связки `&&` и `||`
	  сначала *вычисляется* до конца выражение *слева*, потом, *если нужно*, вычисляется *правый* аргумент
	* тернарная условная операция
	  прежде чем будет вычисляться второй или третий операнд компилятор *вычислит полностью первый операнд*
	* перед вызовом функции всегда завершается *вычисление всех* ее *аргументов* (конкретная последовательность не известна)
	* *операция* "запятая"
	  сначала вычисляется полностью левый аргумент, затем правый
	  не путать с запятой при перечислении аргументов функции
	  см. [[Иные (экзотические) операции#Запятая]]
