## Базовые типы целых чисел (зависят от системы)
Встроенные базовые типы зависят от разрядности слова в целевой системе, потому могут варьироваться и давать неожиданные результаты при переносе программы (потенциально не являются *переносимыми*). 

Тот же тип `int` может иметь различную разрядность в зависимости от ОС. (2 или 4 байта)

### Разрядность и знаковость по типам
Для указания разрядности (выделяемого количество памяти под соответствующие значения) используются следующие ключевые слова:
* `char`  -- обычно 1 байт
* `short` -- обычно 2 байта
* `int` -- обычно 4 байта
* `long` -- обычно 8 байт (Unix), 4 байта (Windows)
* `long long` -- обычно 8 байт

Для указания знаковости используются ключевые слова:
* `signed`
* `unsigned`

#### Тип `char` (`signed` и `unsigned`)
Тип `char` имеет *наименьшую* разрядность: в большинстве случаев это -- ***1 байт***
По другому:
Тип `char` на *любой* платформе равен по размеру *минимальному адресуемому* (минимальная область памяти, которая может иметь собственный адрес)

> [!NOTE] Знаковость типа `char`
> Тип `char` единственный, без указания знаковости явно, имеет неопределенные границы:
> * либо от `0` до `255`
> * либо от `-128` до `127`
> что может зависеть от компилятора, например. По этому с типом `char` следует всегда *явно* указывать знаковость. 
> 
> Дело в том что на момент создания данный тип предназначался для хранения кода символа, а существовала лишь ASCII таблица из 128 позиций, которая прекрасно покрывается обоими вариантами знаковости, так что знаковость типа `char` была не так уж важна
> 
> Остальные типы по умолчанию *знаковые*. 
> Ключевое слово `signed` служит только для типа `char`, в остальном смысла не имеет

#### Тип `short` и `unsigned short`
В языке C нет конкретных требований к разрядности для типа `short` (только то, что он должен быть не меньше `char`).
На всех машинах в реальной жизни -- ***2 байта***.

> [!NOTE] Историческая справка
> Изначально ключевое слово `short` предполагалось в роли *модификатора* как `signed` и `unsigned`. Так можно делать и до сих пор:
> ```
> unsigned short int x;
> ```
> От данного стиля отказались, ибо тут нечему еще быть "коротким" кроме целого числа

#### Тип `int` и `unsigned int`
Разрядность типов `int` и `unsigned int` зависит от архитектуры (и вероятно, компилятора):
* на 16-битных архитектурах был равен `short`, то есть ***2 байта***
* на 32-битных архитектурах равен `long`, то есть ***4 байта***
* на 64-битных архитектурах остался равен ***4 байта***

#### Тип `long` и `unsigned long`
Разрядность типов `int` и `unsigned int` зависит от архитектуры (и вероятно, компилятора):
* до 64-битных архитектур всегда ***4 байта***
* на 64-битных архитектурах зависит от имплементации:
	* ***8 байт*** на Unix-подобных ОС
	* ***4 байта*** на Windows

Существует неофициальное соглашение, согласно которому разрядности типа `long` на любой архитектуре должно хватать для представления указателей (очевидно, этого не придерживается Microsoft)

#### Тип `long long` и `unsigned long long`
Поддерживается не всеми компиляторами и не всеми платформами, но везде разрядность составляет ***4 байта***

### Выводы о разрядности
В ANSI C ситуация такова, что однозначно констатировать можно лишь следующие факты:
* `short` не будет короче `char`
* `int` не будет короче `short`
* `long` не будет короче `int`
* `long long` не будет короче `long`
все остальное потенциально может зависеть от компилятора и/или платформы

## Тип `size_t`
В языке C имеется механизм создания псевдонимов `typedef`, который позволяет указать синонимы для существующих типов. Так например:
```c
typedef double real;
```
сделает возможным указание `real` вместо `double` там, где указывается последний.

Тип `size_t` является целочисленным без знака. Именно такоей тип возвращает операция `sizeof`. Тип `size_t` является синонимом, только определяет его соответствие одному из существующих типов компилятор, подставляя нужный тип из базовых на основании используемой машины/системы.

## Переносимые типы
Стандарты C99 и C11 определяют категории альтернативных имен во избежание проблем с переносимостью, размерность которых не зависит от системы.

### Примеры переносимых типов
#### Типы с точной шириной
Стандарт C99 определяет в заголовочном файле `stdint.h` *целочисленные типы с точной шириной*. Фактически, вводимые имена являются псевдонимами имеющихся базовых типов, для которых компилятор в процессе работы будет выполнять замену на соответствующий по размерности тип ОС.

Так, для одной системы тип с точной шириной `int32_t` будет заменен на `int`, а для другой, может быть заменен на `long`. Однако, потенциально, система может в принципе не поддерживать определенные типы с точной шириной (какой-нибудь `int8_t` на современных ОС просто не будет определен).

По этому типы с точной шириной являются *необязательными* и их поддержка не гарантируется.

#### Типы с минимальной шириной
Являются *обязательным* (к поддержке со стороны системы).

*Тип с минимальной шириной* гарантирует, что тип достаточно велик, чтобы удовлетворить спецификации, и нет иных типов, которые могут выполнить нужную работу, но имеют меньший размер.

Пример -- тип `int_least8_t`, который является *псевдонимом* наименьшего доступного типа, который может хранить 8-битное целочисленное число со знаком.
Так, тип с точной шириной `int8_t` на 16-битной ОС не был бы определен, в то время как тип `int_least8_t` был бы доступным (и вероятно представлял бы собой 16-битный целочисленный тип).

#### Высокоскоростные типы с минимальной шириной
Определяет набор типов, обеспечивающих максимально быстрые вычисления для данных указанного размера. 
Так, `int_fast8_t` определяется как альтернативное имя для целочисленного типа данных, обеспечивающего высокоскоростные вычисления для 8-битных числе со знаком.
Не всегда конечный тип будет при этом совпадать с предыдущей категорией псевдонимов. В одном упор сделан на расход памяти, в другом на быстродействие.

#### Максимально возможный целочисленный тип
Тип определяется именем `uintmax_t`, может хранить любое допустимое целочисленное со знаком (в том числе оно может выходить за рамки `long long`, зависит от компилятора).

### Форматированный ввод и вывод переносимых типов
Для ввода и вывода, как известно, необходимо корректно указывать спецификаторы типа данных. С переносимыми типами доподлинно неизвестно, как именно компилятор определит итоговый тип.

Чтобы решить проблему вывода существует заголовочный файл `inttypes.h`, предоставляющий строковые макросы для отображения переносимых типов. 
Так, макрос `PRId32`определит подходящий спецификатор для вывода 32-битного числа со знаком.
Пример:
```c
#include <inttypes.h>
...
int32_t me32; /* 32-битная переменная со знаком */
me32 = 45933945;
printf("me32 = %" PRId32 "\n", me32);
...
```
Заголовочный файл `inttypes.h` включает в себя вызов заголовочного файла `stdint.h`

Вывод рабочей тестовой программы:
```
15:43 $ ./altnames    
Сначала предположим, что int32_t является int: me32 = 45933945  
Далее не будем делать никаких предположений.  
Вместо этого воспользуемся "макросом" из файла inttypes.h: me32 = 45933945
```
